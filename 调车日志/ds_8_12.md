# 编码器调试记录 — 方向编码器 vs 正交编码器

> 日期：2026-06-12
> 问题：motor_test 电机转动但编码器读数为 0

---

## 问题排查过程

### 1. 现象
- motor_test 菜单中电机可以转动
- 但 IPS 屏幕和串口终端显示 `L_speed=0`, `R_speed=0`
- `xxd -l 2 /dev/zf_encoder_dir_1` 始终返回 `0000`

### 2. 排查步骤

| 步骤 | 操作 | 结果 |
|------|------|------|
| ① 怀疑延迟不够 | `motor_speed_test` 中加 `system_delay_ms(10)` | 仍为 0 |
| ② 怀疑编码器没清零 | `main.cpp` 加 `encoder1/2.clear_count()` | 仍为 0 |
| ③ 连续读 3 次不清零 | 三次 `get_count()` 看值是否增长 | 全为 0 |
| ④ 手动读设备文件 | `xxd -l 2 /dev/zf_encoder_dir_1` | 始终 0000 |
| ⑤ 查设备树 | `seekfree_2k0300_coreboard.dts` | **找到根因！** |

### 3. 根因

设备树中启用了**正交编码器** (`encoder_quad`)，但实际硬件是**方向编码器** (`encoder_dir`)：

```dts
// ❌ 当前状态：正交编码器已启用
&encoder_atim{
    status = "okay";
    compatible = "seekfree,encoder_quad";   // ← 正交模式
    pinctrl-0 = <&atim_ch1_pin_m0>, <&atim_ch2_pin_m0>;
};

// ✅ 需要改成：方向编码器
&encoder_atim{
    status = "okay";
    compatible = "seekfree,encoder_dir";    // ← 方向模式
    pinctrl-0 = <&atim_ch1_pin_m0>;
    dir-gpios = <&gpa1 13 GPIO_ACTIVE_LOW>;
};
```

正交编码器和方向编码器**共享同一块硬件**（`encoder_atim` / `encoder_gtim`），只能二选一。内核按正交模式操作硬件，与方向编码器硬件不匹配，所以脉冲计数始终为 0。

## 方向编码器引脚定义

| 编码器 | 脉冲信号脚 | 方向脚 | 设备文件 |
|--------|-----------|--------|----------|
| 右轮 (ATIM) | **GPA1 pin 12** | **GPA1 pin 13** | `/dev/zf_encoder_dir_1` |
| 左轮 (GTIM) | **GPA2 pin 2** | **GPA2 pin 3** | `/dev/zf_encoder_dir_2` |

---

## 两种编码器模式对比

| | 正交编码器 (quad) | 方向编码器 (dir) |
|---|---|---|
| 信号线 | 2 根 (A+B相, 90°相位差) | 1 根脉冲 + 1 个 GPIO 方向 |
| 方向判断 | 硬件自动 (相位比较) | 软件 (读 GPIO 电平) |
| 精度 | 4倍频 (双边沿计数) | 1倍频 |
| compatible | `"seekfree,encoder_quad"` | `"seekfree,encoder_dir"` |
| 设备文件 | `/dev/zf_encoder_quad_*` | `/dev/zf_encoder_dir_*` |

## 什么是设备树 (Device Tree)

设备树（`.dts`）是一个硬件描述文件，告诉 Linux 内核：
- 板子上有哪些外设
- 它们接在哪些引脚上
- 用什么驱动

编译后生成 `.dtb` 二进制文件，内核启动时加载。

```
.dts (源码) ──dtc编译──→ .dtb (二进制) ──内核加载──→ 创建设备文件
```
